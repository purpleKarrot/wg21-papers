# Abstract

`std::ranges::view_interface` provides `back()` only for views that model
`bidirectional_range` and `common_range`. That is sufficient to specify
`back()` as `*ranges::prev(ranges::end(r))`, but it is not the only case in
which `back()` can be provided efficiently.

A view that models `random_access_range` and `sized_range` can also provide
`back()` in constant time, even if it does not model `common_range`. This paper
extends `view_interface::back()` to that case.

# Motivation

`view_interface` is intended to give views a familiar container-like interface.
It already provides `front()` for `forward_range`, `back()` for
`bidirectional_range && common_range`, and `operator[]` for
`random_access_range`.

That leaves a gap: a view can be random-access enough to support indexing and
sized enough for `ranges::size(r)` to be available, yet still not receive
`back()` solely because its sentinel is distinct from its iterator.

The `common_range` requirement is not fundamental to `back()` itself. It is
only needed for the current specification strategy, which computes the last
iterator as `ranges::prev(ranges::end(r))`. For a `random_access_range` that is
also a `sized_range`, there is another constant-time formulation:

> `ranges::begin(r)[ranges::size(r) - 1]`

That formulation does not require `common_range`.

# Proposal

Extend the constraints on the two `view_interface::back()` overloads to also
permit `random_access_range && sized_range`.

This does not change semantics for the existing common-bidirectional case. It
makes additional programs well-formed by enabling `back()` for sized
random-access views whose iterator and sentinel types differ.

# Design considerations

## Why not all bidirectional ranges?

This paper does not propose `back()` for arbitrary non-common bidirectional
ranges. Without `common_range`, the current `ranges::prev(ranges::end(r))`
formulation is unavailable, and bidirectional traversal alone does not provide a
separate constant-time route to the last element.

By contrast, `random_access_range && sized_range` does provide such a route.

## Why `ranges::size`?

The newly-enabled case is justified precisely by `sized_range`, so the wording
should be phrased in terms of `ranges::size`.

It is also important to use `ranges::size(r)` rather than the member
`view_interface::size()`. The member is available only under its own
constraints, whereas `ranges::size(r)` is the primitive operation guaranteed by
`sized_range`.

## Complexity

The operation remains constant time in all supported cases.

# Impact on existing code

This is an additive change. Existing well-formed programs are unaffected.
Expressions that are currently ill-formed become well-formed for the newly
supported category of views.

Code that explicitly checks whether a view has a `back()` member may therefore
observe different results. That is the intended effect.

# Wording

The wording below is relative to the current working draft.

In the synopsis in [view.interface.general]{.sref}, replace the declarations of
`back()` with:

```cpp
constexpr decltype(auto) back() requires
  (bidirectional_range<D> && common_range<D>) ||
  (random_access_range<D> && sized_range<D>);
constexpr decltype(auto) back() const requires
  (bidirectional_range<const D> && common_range<const D>) ||
  (random_access_range<const D> && sized_range<const D>);
```

In [view.interface.members]{.sref}, replace the declarations of `back()` with:

```cpp
constexpr decltype(auto) back() requires
  (bidirectional_range<D> && common_range<D>) ||
  (random_access_range<D> && sized_range<D>);
constexpr decltype(auto) back() const requires
  (bidirectional_range<const D> && common_range<const D>) ||
  (random_access_range<const D> && sized_range<const D>);
```

The precondition remains unchanged:

> `!empty()` is `true`.

Replace the *Effects* element with:

Equivalent to:

```cpp
if constexpr (common_range<decltype($derived$())>)
  return *ranges::prev(ranges::end($derived$()));
else
  return ranges::begin($derived$())[ranges::size($derived$()) - 1];
```
